## 0. 一句话结论

这次口头确认实质上是一次**产品主线切换**：从「C 端微信小程序（7 天陪伴周期）」切到「B 端 AI 私域运营 SaaS」。仓库里现有的生产架构文档是为前者写的——后端架构决策大部分可以平移，但客户端形态、核心引擎、MVP 路径都要按新方向重排。最大的技术不确定性不在 AI，在**微信群数据怎么合规地进系统**。

## 2. 现有资产盘点与复用矩阵

### 2.1 直接平移（价值高，不用重做决策）

- **后端架构决策**：TypeScript + NestJS + PostgreSQL + Prisma、typed REST、腾讯云优先但 SDK 隔离在 adapter 层、server-side AI gateway、DeepSeek 首选国产模型、事件模型（`user.created` 等换成 `group.imported` / `reminder.generated` 等即可）、备份纪律。这套决策对 B 端 SaaS 同样成立，而且 adapter 边界对 SaaS 更重要。
- **CRM 标签数据原则**（§8）：标签结构化存储、区分来源（显式/行为/AI 推断+置信度/人工）、带时间戳——这几乎就是**私域健康看板的数据模型底座**，原样可用。
- **AI 红线**（§6）：AI 不得输出医疗/心理宣称、不做业务决策——B 端内容生成必须继承这条（大健康赛道的合规命脉）。

### 2.2 部分复用

- **demo 分支**（15 个 commit，Next.js，module-01~12）：定位是 C 端 demo（已归档原型），但它的 web 技术栈、设计系统、Playwright 冒烟经验与 B 端管理台同构。**建议作为经验参考，不作为代码基座**（架构文档明确 demo 不做生产基础，这条对新方向依然对）。
- **海报/内容管线**（§7）：文本模型生成结构化内容 → 素材 → 服务端渲染 → 对象存储。内容日历模块的"生成朋友圈图文素材"可以直接复用这个管线设计。

### 2.3 本方向下冻结（不是删除，是不投入）

- 微信小程序客户端决策（TDesign/Vant 选型等 pending 项全部冻结）
- 7 天周期引擎、C 端 intake 问题集、打卡/最终报告流
- FOUNDER_NON_DEV_ACTION_PLAN 里的小程序注册/支付路径项（域名/备案/AI 账号等仍有用）

## 3. 三模块技术拆解（按实现难度排序）

### 模块 3：内容日历生成 —— 最易，建议первый做
- 输入：品牌行业 + 月度主题（One Day 方法论）→ LLM 生成 7 天朋友圈内容 + 群话题 + 私信话术包。
- **零数据依赖**、纯生成，prompt 工程 + 品牌配置表即可。1-2 周出可演示版本。
- 风险仅在内容质量与合规过滤（禁医疗宣称词表）。

### 模块 2：智能互动提醒 —— 中等
- 本质：成员级"上次互动时间线" + 规则引擎（"7 天未互动 → 触发"）+ LLM 按上下文生成话术建议 → 每日行动清单。
- 冲刺文档自己也说了：**MVP 不用真模型训练，规则引擎 + prompt 就够**。难点不在 AI，在拿到"谁在什么时候说了什么"的数据（见 §4 关键路径）。

### 模块 1：私域健康看板 —— 最重
- 分层（活跃/沉睡/高意向/流失）+ 私域健康评分。依赖完整群数据管道 + "健康分"的口径定义（需要客户的运营 know-how 编码成规则）。
- 可视化本身不难，难在数据完备性和口径可信度。MVP 版建议做成"分层列表 + 几个关键计数"，不做炫图表。

### 4. 共同关键路径：微信群数据接入（最大技术+合规不确定性）

三条路线：

| 路线 | 说明 | 判断 |
|---|---|---|
| A. 群聊记录导出 → 手动上传解析 | 品牌方导出聊天记录文件，上传到系统解析 | **MVP 现实解**。冲刺文档的模块定义（"输入：微信群聊天记录导出"）就是这个假设。缺点：非实时、依赖人工操作 |
| B. 企业微信官方 API | 会话存档 API（需企业开通+成员同意）、客户联系 API | **合规正道、长期方向**。门槛：品牌必须用企微且开通会话存档（付费）；个人隐私同意机制内建 |
| C. 个人微信自动化（外挂/协议） | hook 个微收发消息 | **不做**。违反微信 ToS，封号+法律风险，对 SaaS 是致命依赖 |

建议：MVP 走 A，pilot 谈判时把"是否用企微"作为品牌筛选条件之一，v1.0 上 B。

## 4. 风险清单（按严重度）

1. **数据合规（PIPL）**：群成员聊天记录是个人信息，品牌把它交给第三方（我们）处理，需要合法性基础（告知/同意或匿名化）。pilot 协议必须含数据处理条款；系统入库前做手机号/敏感信息脱敏。这条不解决，投资人尽调也过不去。
2. **微信生态 ToS**：坚决不碰个微自动化；对外宣传口径也别暗示"自动私发消息"——工具输出的是**给运营人员的行动建议**，发送动作由人完成（这同时规避了合规和"AI 乱发话术"的品牌风险）。
3. **大健康宣称合规**：AI 生成内容禁医疗/疗效宣称（"治好""根治"等词表过滤 + 人工复核位）。架构文档 §6 的红线原样继承。
4. **pilot 效果归因**：互动率提升受运营执行影响大，AI 价值难剥离。pilot 设计必须有 baseline（前 30 天）对照 + 选"有基础但没运营好"的群（冲刺文档建议正确）。
5. **两线作战**：C 端小程序与 B 端 SaaS 不可能并行开发。需要明确冻结小程序（见开放问题 1）。
6. **"10 倍降本"表述**：9800/年 vs 运营岗 8-12 万/年数学成立，但 MVP 阶段工具是"辅助"不是"替代"。对品牌和投资人的表述建议是"替代 XX% 重复劳动"，留出余地，避免 pilot 后预期落差。
7. **demo 级 vs 真用级混淆**：hackathon/投资人 demo 可以宽松，品牌 pilot 拿真实群数据就必须有基本的数据安全（加密存储、访问控制、删除机制）。两个目标别用同一套宽松标准。

## 5. MVP 切法建议

**Phase 0（约 1-2 周）｜可演示核**
- 内容日历生成器（模块 3 完整版）+ 群聊记录解析器（路线 A，打数据底座）
- 产出：能现场演示"上传一份群记录 → 看到成员列表和基础统计 → 生成一周内容包"

**Phase 1（约 3-6 周）｜pilot 可用版**
- 互动提醒引擎（规则 + LLM 话术）+ 简版健康看板（分层列表 + 健康分 v0）
- 每日行动清单页（运营人员的工作台）
- baseline/pilot 数据对比报表（这是 pilot 成败的证据链，值得当功能做）

**技术栈**：沿用已定架构决策（NestJS + Postgres + Prisma + DeepSeek adapter + 对象存储 adapter），前端 Next.js Web 管理台。不做小程序、不做支付、不做多租户隔离的完整版（pilot 期单/双租户简化）、不训练自有模型。

**相对冲刺文档的一个修正**：文档建议非技术创始人先用 Dify/Coze 低代码出 v0.1。我们有 agent 开发团队，**可以直接走代码 MVP（文档里的"路径 A"），跳过低代码阶段**——同样 2-3 周拿到的是可迭代的真产品而非一次性原型。文档里"最缺技术合伙人"的风险，当前团队结构（主开发 + Reviewer + 协调）已部分对冲。
